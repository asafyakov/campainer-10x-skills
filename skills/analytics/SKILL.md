# סקיל: analytics — ניתוח קמפיין והמלצות

## מה הסקיל הזה עושה

שולף נתונים מ-Facebook API + מערכת CRM/דיוור (רב מסר, ActiveCampaign, Monday, HubSpot, או כל מערכת עם API) במקביל, מציג דוח אחד משולב, ועובר עץ אבחון מלא לפי שיטת אסף יעקב.
**מערכת ה-CRM = מקור האמת לכמות לידים.** פייסבוק = מקור לעלויות, CTR, CPM.

> **הערה:** הדוגמאות בסקיל הזה כתובות עם רב מסר (Responder) כמימוש ברירת מחדל, אבל הלוגיקה זהה לכל מערכת CRM/דיוור עם API. אם הלקוח עובד עם מערכת אחרת — התאם את קריאות ה-API בהתאם.

24–72 שעות אחרי שהקמפיין עלה לאוויר.
אם שואלים "מה התוצאות?" / "כמה לידים?" / "שווה להמשיך?" — הפעל אוטומטית.

**אפשרות א' — חיפוש מקומי:**
חפש קבצי `campaign_id.txt` בתת-תיקיות `*-fb/`:

```bash
find . -name "campaign_id.txt" -path "*-fb/*"
```

- נמצא אחד → קרא את ה-ID, עבור לשלב 2
- נמצאו כמה → שאל: "על איזה קמפיין לבדוק?"

**אפשרות ב' — שליפה מהחשבון:**

```python
r = requests.get(
    f"https://graph.facebook.com/v22.0/{AD_ACCOUNT_ID}/campaigns",
    params={
        "access_token": ACCESS_TOKEN,
        "fields": "name,status,start_time,objective",
        "effective_status": '["ACTIVE","PAUSED"]',
        "limit": 20
    }
)
```

> **חשוב:** לא להשתמש ב-`filtering` — לא נתמך. להשתמש ב-`effective_status`.

## שלב 1.5 — שאלון מהיר (חוסם — חובה לפני שליפת נתונים)

> **⛔ אסור לשלוף נתונים מרב מסר / מערכת דיוור לפני שעברת את כל השאלות כאן ויש לך תשובות מפורשות.**
> גם אם יש רשימה עם שם דומה לקמפיין — לא להניח שהיא הנכונה.

שאל 3 שאלות ברצף:

> 1. "מה סוג הקמפיין?
>    א. מגנט לידים / וובינר / הדרכה חינמית
>    ב. לידים לתוכנית / ליווי / שירות פרימיום
>    ג. מכירות (מוצר פרונט) — מה המחיר?"
>
> 2. "הלידים נאספים בתוך פייסבוק (טופס לידים / וואטסאפ) או דרך דף נחיתה חיצוני?"
>
> 3. **אם דף נחיתה חיצוני:** "לאן הלידים נכנסים? רב מסר? מערכת אחרת? ומאיזו רשימה ספציפית לשלוף?"

**תוצאות:**
- לידים בתוך פייסבוק → נתוני פייסבוק מדויקים, **לדלג על הצלבה לגמרי**
- דף נחיתה חיצוני + רב מסר → חייבים הצלבה. **לשלוף רק מהרשימה שהמשתמש אישר**
- דף נחיתה חיצוני + מערכת אחרת → לשאול איך לגשת אליה

> **שמירת תשובות:** אחרי שהמשתמש ענה על השאלות — שמור את התשובות בסקשן "הגדרות analytics" ב-CLAUDE.md:
> ```markdown
> ## הגדרות analytics
> - סוג קמפיין: [מגנט/פרימיום/מכירות]
> - איסוף לידים: [פייסבוק/דף חיצוני]
> - CRM: [רב מסר/אחר/אין]
> - רשימה/מקור: [שם הרשימה]
> ```
> בפעם הבאה שמריצים analytics — בדוק אם הסקשן קיים. אם כן — **דלג על השאלון** ותשתמש במה שכבר שמור. אם המשתמש רוצה לשנות — "תגיד 'עדכן הגדרות' ואני אשאל מחדש."

## שלב 2 — שליפת מטריקות

### שליפה אחת מפייסבוק — עם `time_increment=1`

**קריאת API אחת בלבד.** שולפים ברמת מודעה עם `time_increment=1` — מקבלים שורה לכל יום לכל מודעה. מזה מחשבים כל טווח שרוצים בלי לפנות שוב ל-API.

```python
import sys, json, datetime
from pathlib import Path
sys.path.insert(0, str(Path.home() / ".claude"))
from fb_config import ACCESS_TOKEN  # ⛔ רק TOKEN — AD_ACCOUNT_ID מגיע מ-CLAUDE.md
import requests

CAMPAIGN_ID = "[ID]"
TODAY = datetime.date.today().isoformat()
# start_date = תאריך התחלת הקמפיין (מ-start_time)
TIME_RANGE = json.dumps({"since": start_date, "until": TODAY})

r = requests.get(
    f"https://graph.facebook.com/v22.0/{CAMPAIGN_ID}/insights",
    params={
        "access_token": ACCESS_TOKEN,
        "fields": "ad_name,adset_name,ad_id,adset_id,impressions,inline_link_clicks,spend,actions,cpm,reach,frequency",
        "time_range": TIME_RANGE,
        "level": "ad",
        "time_increment": 1,
        "limit": 500
    }
)
```

> **יתרונות:** קריאה אחת במקום 12+. אפשר לפרק לכל טווח שרוצים. קל על המחשב.

> **חשוב:** `last_7d` / `last_14d` לא כוללים את היום הנוכחי. תמיד להשתמש ב-`time_range` עם תאריכים מפורשים.

### טווחים דינמיים לפי גיל הקמפיין

חשב את גיל הקמפיין (ימים מ-start_date עד היום) ובחר טווחים:

| גיל קמפיין | טווחים |
|-------------|--------|
| **עד 5 ימים** | מקסימום, 3 ימים, אתמול |
| **6–14 ימים** | מקסימום, 7 ימים, 3 ימים |
| **15–30 ימים** | מקסימום, 14 ימים, 7 ימים, 3 ימים |
| **30+ ימים** | מקסימום, 30 ימים, 14 ימים, 7 ימים, 3 ימים |

**חישוב כל טווח:** מסננים את השורות היומיות לפי תאריך וסוכמים: spend, clicks, leads, impressions, lpv. מחשבים CPC, CPL, CPM, CTR, המרה מהסכומים.

### שליפה מרב מסר (במקביל)

**אם יש דף נחיתה חיצוני ויש חיבור לרב מסר:**

```python
# קבלת JWT
r_token = requests.post("https://graph.responder.live/v2/oauth/token",
    json={
        "grant_type": "client_credentials",
        "scope": "*",
        "client_id": 163,
        "client_secret": RAVMESAR_CLIENT_SECRET,
        "user_token": RAVMESAR_USER_TOKEN
    }, timeout=15)
jwt = r_token.json()["token"]

# שליפת נרשמים מהרשימה הרלוונטית — עם דה-דופליקציה
all_subs = []
seen_ids = set()
page = 1
while page <= 50:
    r_subs = requests.get(
        f"https://graph.responder.live/v2/lists/{LIST_ID}/subscribers",
        headers={"Authorization": f"Bearer {jwt}"},
        params={"page": page, "limit": 500},
        timeout=15
    )
    data = r_subs.json().get("data", [])
    if not data:
        break
    
    new_count = 0
    for s in data:
        sid = s.get("id")
        if sid and sid not in seen_ids:
            seen_ids.add(sid)
            all_subs.append(s)
            new_count += 1
    
    # אם כל הרשומות בדף הזה כבר נראו — עצור (API מחזיר דופליקטים)
    if new_count == 0:
        break
    
    page += 1
```

> **⚠️ באג ידוע ב-API של רב מסר:** הפאגינציה יכולה להחזיר את אותם נרשמים שוב ושוב בכל דף.
> **חובה** לעשות דה-דופליקציה לפי `id` ולעצור ברגע שדף מחזיר 0 רשומות חדשות.
> **בדיקת שפיות:** לאחר השליפה, להשוות את הכמות שנשלפה מול מספר הנרשמים שהרשימה מדווחת (subscribers_count או מה שרואים בממשק). אם יש פער גדול — יש באג בשליפה.

**זיהוי הרשימה הנכונה:**
1. **לא להניח** שרשימה עם שם דומה היא הנכונה — לשאול את המשתמש מפורשות
2. שלוף את כל הרשימות: `GET /v2/lists` — להציג למשתמש אם צריך
3. רק אחרי אישור מפורש — לשלוף נרשמים

**שדות חשובים בכל subscriber:**
- `id` — מזהה ייחודי (**חובה לדה-דופליקציה**)
- `list_created` — מתי נוסף **לרשימה הזו** (התאריך הרלוונטי לסינון)
- `personal_fields.96444` (שדה "ref") — שם המודעה ששלחה אותו (אם מוגדר בדף)
- `email`, `name`, `phone` — פרטים
- `blocked` — 1 = חסום (לא לספור)

**סינון:**
- ספור רק subscribers ש-`list_created` >= תאריך התחלת הקמפיין
- הוצא `blocked = 1`
- קבץ לפי שדה ref (שם מודעה) להצלבה ברמת מודעה

**בדיקת שפיות אחרי סינון:**
- השווה כמות שנספרה מול מה שרואים בממשק הרשימה
- אם פער > 20% — דווח למשתמש ובדוק את הלוגיקה

## שלב 3 — חישובים

**מטריקות ברמת קמפיין:**
- **לידים רב מסר** = כמות subscribers תקינים ברשימה (מקור אמת)
- **לידים פייסבוק** = `lead` מתוך actions
- **פער** = (רב מסר - פייסבוק) / רב מסר × 100
- **עלות ליד אמיתית** = spend ÷ לידים רב מסר
- **קליקים על קישור** = `inline_link_clicks`
- **Landing Page Views** = `landing_page_view` מתוך actions
- **CPC** = spend ÷ inline_link_clicks
- **CTR (קישור)** = inline_link_clicks ÷ impressions × 100
- **CPM** = מתוך `cpm`
- **המרה בדף** = לידים רב מסר ÷ landing_page_views × 100
- **Frequency** = מתוך `frequency`

**ברמת מודעה — הצלבה עם ref:**
- לכל מודעה: ספור כמה subscribers יש עם ref תואם לשם המודעה
- אם אין ref → ציין "ללא ref"

## שלב 4 — עץ אבחון

### יעדים לפי סוג קמפיין

**מגנט לידים / וובינר / הדרכה חינמית:**
- עלות ליד (אמיתית): עד 15 ש"ח ✅ | 15–20 ש"ח ⚠️ | מעל 20 ש"ח ❌
- המרה בדף: 20–30% ✅ | מתחת ⚠️
- CTR: מעל 2% ✅ | 1.5–2% ⚠️ | מתחת 1.5% ❌

**לידים לתוכנית / ליווי / פרימיום:**
- עלות ליד (אמיתית): עד 100 ש"ח ✅ | 100–150 ש"ח ⚠️ | מעל 150 ש"ח ❌
- המרה בדף: 10–15% ✅ | מתחת ⚠️
- CTR: מעל 2% ✅ | 1.5–2% ⚠️ | מתחת 1.5% ❌

**מכירות — מוצר פרונט:**
- עלות רכישה: עד 90–100 ש"ח ✅ | עד חצי מחיר ⚠️ | מעל ❌
- המרה בדף: 4–10% ✅ | מתחת ⚠️
- CTR: מעל 2% ✅ | 1.5–2% ⚠️ | מתחת 1.5% ❌

**לכל הסוגים:**
- CPM: מתחת 30 ✅ מצוין | 30–80 ✅ תקין | מעל 80 ⚠️
- CPC: עד 4 ש"ח ✅ | מעל 4 ⚠️

### אבחון: מודעה או דף?

```
עלות תוצאה גבוהה
  ↓
CPC מעל 4 ש"ח?
  ├─ כן → בעיה במודעה (שלב 5)
  └─ לא → בעיה בדף (שלב 6)
```

## שלב 5 — אבחון מודעה

### CPM מעל 80?
- Frequency מעל 3–4 → קהל עייף → הרחב קהל / עצור
- עונתיות → המתן
- Frequency תקין → קריאייטיב חלש → שנה הוק

### חלוקת תקציב
- מודעה אחת 70%+ מהתקציב + תוצאות רעות → כבה ותן צ'אנס לשאר
- מודעה אחת 70%+ + תוצאות טובות → בסדר, פייסבוק מיטב

### סף שיפוט (דינמי)

**הסף הוא דינמי — לא מספר קבוע:**

1. **אין דאטה על הקמפיין כלל** (קמפיין חדש, אין CPL ידוע):
   - מגנט לידים → מינימום 50 ש"ח
   - פרימיום / מוצר → מינימום 100 ש"ח

2. **יש דאטה על הקמפיין** (CPL ידוע, מודעות אחרות כבר רצות):
   - אפשר להתאים את הסף לפי ה-CPL של הקמפיין
   - **אבל לעולם לא מתחת ל-20–30 ש"ח** — המודעה חייבת מספיק חשיפה (impressions/reach) כדי שהאלגוריתם יוכל לאופטימז

3. **כללים:**
   - מתחת לסף → ⏳ ממתין, **אסור להמליץ לכבות**
   - CPC גבוה על הוצאה נמוכה = רעש סטטיסטי, לא מגמה
   - מעל הסף + לא עומד ביעד → חבוט
   - 3–4 מודעות באותה זווית כולן נכשלות **ועברו את הסף** → כבה ad set + זווית
   - **לעולם לא לכבות ad set שלם בלי לבחון כל מודעה בנפרד** — מודעה חלשה לא הורגת זווית

## שלב 6 — זיהוי בעיה בדף נחיתה (הפנייה בלבד)

```
CPC תקין + המרה נמוכה
  ↓
  הבעיה בדף הנחיתה, לא במודעה.
```

**הסקיל הזה לא מאבחן דפי נחיתה.** הוא רק מזהה שהבעיה שם.

הצג למשתמש:
> "הנתונים מראים ש-CPC תקין ([X]₪) אבל המרה בדף נמוכה ([X]%).
> זה אומר שהמודעות עובדות — אנשים לוחצים — אבל הדף לא ממיר.
> 
> רוצה לבדוק את הדף? הרץ `/page-cro` עם URL הדף."

**אסור לסקיל הזה:** לנתח תוכן דף, להמליץ על שינויי כותרת/CTA, לעשות אופטימיזציית דף. זה תפקיד של `/page-cro`.

## שלב 7 — מתי לכבות / לסקייל

**כיבוי:**
- מגנט: CPL מעל 20 ש"ח אחרי אופטימיזציה
- פרימיום: לא רווחי אחרי כמה זוויות
- 3–4 זוויות נכשלו → בדוק דף/הצעה/קהל מהבסיס

**סקיילינג (קמפיין עובד):**
1. העלאת תקציב 10–20% כל 2–3 ימים
2. שכפול ad set עם מודעות חדשות
3. CBO עם ווינרים (אחרי 2–4 שבועות)

**⛔ כלל יציבות תקציב — לפי ad set (לא כללי!):**

הכלל חל **על כל ad set בנפרד**, לא על כל הקמפיין. אפשר לשנות תקציב ל-ad set אחד בזמן שאחר "קפוא".

1. **לפני כל המלצת תקציב** → בדוק ב-learnings.md מתי היה שינוי אחרון **ל-ad set הספציפי הזה**
2. **אם היה שינוי ב-3-5 ימים האחרונים לאותו ad set** → לא להמליץ. לכתוב: "⏳ [שם ad set]: שינוי תקציב אחרון ב-[תאריך]. ממתין ליציבות."
3. **אסור שינוי הפוך לאותו ad set:** אם הורדנו → לא מעלים עד 3-5 ימים עם דאטה חדש שמצדיק. אם העלינו → לא מורידים אלא אם בעיה חריפה.
4. **ad sets אחרים** שלא שינינו להם כלום → **אפשר לשנות** להם תקציב ללא הגבלה.
5. **בדוח:** כשמציגים ad set עם שינוי תקציב אחרון, לציין: "שינוי אחרון: [תאריך] [+/-X%]. לא לשנות עד [תאריך+3-5]."

## שלב 10 — ביצוע פעולות

**רק אם המשתמש מבקש. לא לבצע אוטומטית.**

| פעולה | API |
|-------|-----|
| כיבוי מודעה | `PATCH /{ad_id}` → `{"status": "PAUSED"}` |
| הדלקת מודעה | `PATCH /{ad_id}` → `{"status": "ACTIVE"}` |
| כיבוי ad set | `PATCH /{adset_id}` → `{"status": "PAUSED"}` |
| הדלקת ad set | `PATCH /{adset_id}` → `{"status": "ACTIVE"}` |
| שינוי תקציב | `PATCH /{adset_id}` → `{"daily_budget": "[אגורות]"}` |
| שכפול ad set | `POST /{adset_id}/copies` |
| **מחיקה** | **אסור לעולם** — תמיד כיבוי |

אשר לפני ביצוע. דווח אחרי.

### ולידציה אחרי כל פעולת כתיבה (חובה)

> **⛔ אחרי כל PATCH או POST בשלב הזה — חובה לקרוא GET מה-API ולוודא שהשינוי נכנס כמו שצריך.**

**שכפול אדסט** — הפעולה הכי מסוכנה (באג מתועד: טירגוט נשבר אחרי שכפול):
```python
# אחרי POST /{adset_id}/copies:
new_adset_id = result.get("copied_adset_id")

# חובה: קרא את האדסט החדש מה-API
r = requests.get(f"https://graph.facebook.com/v22.0/{new_adset_id}",
    params={"access_token": ACCESS_TOKEN,
            "fields": "targeting,is_advantage_audience,optimization_goal,bid_strategy,daily_budget,promoted_object,name"})
actual = r.json()

# השווה לאדסט המקורי
r_orig = requests.get(f"https://graph.facebook.com/v22.0/{source_adset_id}",
    params={"access_token": ACCESS_TOKEN,
            "fields": "targeting,is_advantage_audience,optimization_goal,bid_strategy,daily_budget,promoted_object"})
original = r_orig.json()

# בדוק שהטירגוט זהה
if actual.get("targeting") != original.get("targeting"):
    print(f"🚨 טירגוט השתנה אחרי שכפול!")
    print(f"  מקורי: {original.get('targeting', {}).get('geo_locations', {})}")
    print(f"  חדש: {actual.get('targeting', {}).get('geo_locations', {})}")
    # PATCH עם הטירגוט המקורי
    requests.post(f"https://graph.facebook.com/v22.0/{new_adset_id}",
        data={"access_token": ACCESS_TOKEN,
              "targeting": json.dumps(original["targeting"]),
              "is_advantage_audience": str(original.get("is_advantage_audience", False)).lower()})
    print("🔧 תוקן אוטומטית. מוודא שוב...")
    # validate שוב...

if actual.get("is_advantage_audience") != original.get("is_advantage_audience"):
    print(f"🚨 is_advantage_audience השתנה! {original.get('is_advantage_audience')} → {actual.get('is_advantage_audience')}")
    # PATCH...
```

## כללים

1. **לא להמציא מספרים** — רק מה שחזר מה-API
2. **עברית תמיד**
3. **CRM = מקור אמת ללידים** — תמיד להציג ראשון ובבולד
4. **דוח אחד משולב** — לא FB בנפרד ואז הצלבה בנפרד
5. **לעולם לא `last_7d`** — להשתמש ב-`time_range` או `maximum`
6. **המלצה = פעולה קונקרטית** — לא "שקול לשנות"
7. **לעולם לא למחוק** — תמיד כיבוי
8. **אם אין דף נחיתה חיצוני** — לדלג על הצלבת CRM
9. **שגיאות API** — לטפל לבד, לדווח רק אם צריך פעולה
10. **תמיד לבדוק כל הטווחים** — מקסימום, 30d, 14d, 7d, 3d (לפי גיל הקמפיין). לא לקבל החלטה על בסיס טווח אחד
11. **לפני כיבוי מודעה — בדוק מגמה** — מודעה שעובדת על בסיס כל הזמנים לא מכבים בגלל 3 ימים רעים. חייבים לראות מגמה שלילית ב-3d מול 7d מול 14d לפני שמכבים
12. **תמיד מחפש ווינרים** — הדגש הראשון בכל דוח: מה עובד ואיך לסקייל. לא רק "מה לכבות"
13. **עובד לפי הנחיות META** — לא לעשות פעולות שמפרות את מדיניות פייסבוק. לא spam, לא שינויים מאסיביים, לא הפרות
14. **שמירה על החשבון** — לא לעשות 50 שינויים ברגע אחד. פעולה אחת → ולידציה → פעולה הבאה. פייסבוק מגביל API calls ושינויים מאסיביים יכולים לגרום לבעיות בחשבון
15. **סקיל analytics לא מתקן דפי נחיתה** — אם הבעיה בדף, הסקיל מזהה ומפנה. לא מנתח תוכן דף, לא ממליץ על שינויי כותרת/CTA
16. **סקיל analytics לא כותב קופי** — אם צריך קופי חדש, מפנה ל-`/facebook-ad-copywriter`
17. **סקיל analytics לא מייצר קריאייטיבים** — אם צריך, מפנה ל-`/creative-ai` או `/creative-ai`
